## 1. 项目概述

**Vineyard Mower** 是一个面向葡萄园环境的自主割草机器人系统。系统负责协调感知、导航、运动控制和任务规划等子系统，在结构化葡萄园环境中实现无人化割草作业。

### 1.1 核心目标

| 目标 | 优先级 | 衡量标准 |
|------|--------|----------|
| 安全作业 | **P0** | 零人员伤害、零藤蔓损伤 |
| 全覆盖割草 | **P1** | 行间覆盖率 ≥ 95% |
| 自主导航 | **P1** | GPS/RTK 不可用区域正常运行 |
| 地形适应 | **P2** | 坡度 ≤ 25° 可稳定作业 |
| 作业效率 | **P3** | 不低于人工割草效率的 80% |

### 1.2 环境假设

- **场地类型**: 结构化葡萄园（行栽作物）
- **行距**: 2.0–3.5 m（可配置）
- **行内障碍**: 藤蔓主干、立柱、滴灌管线
- **坡度范围**: 0°–25°（>15° 进入减速模式）
- **土壤条件**: 干燥到半湿润（湿润触发扭矩异常检测）
- **GPS 条件**: RTK-GPS 为主，冠层遮挡区域退化到视觉/激光里程计
- **人员存在**: 作业区域可能存在工作人员，必须实时检测并响应

### 2.1 子系统职责

| 子系统 | 职责 | 安全等级 |
|--------|------|----------|
| **感知 (Perception)** | 激光/视觉/RTK 融合；障碍物检测；行/棚架识别；人员检测 | **安全关键** |

| **数据管道 (Data Pipeline)** | 遥测日志；地图管理；作业历史；性能分析 | 非关键 |

## 3. 安全原则 (Safety Invariants)

> **所有涉及物理执行（电机、舵机、刀盘）的指令，必须在执行前经过运动控制子系统的确定性验证。**  
> **当传感器数据与规划路径冲突时，传感器数据为准——世界是 ground truth。**

### 3.1 运行时安全不变量清单

每次启动及运行中持续验证：

- [ ] 所有急停按钮功能正常
- [ ] 所有电子围栏已激活
- [ ] 碰撞避免系统在线
- [ ] 人员存在检测处于活动状态
- [ ] 刀盘保护罩在位
- [ ] 通信链路延迟 < 500ms（自主模式）/ < 200ms（遥控模式）

### 3.2 安全行为

| 事件 | 响应 | 恢复条件 |
|------|------|----------|
| 检测到人员进入警戒区 | 立即停止 → 上报 → 等待 | 操作员显式解除 |
| 检测到不可分类障碍物 | 停止 → 分类尝试 → 上报 | 操作员确认安全 |
| GPS/RTK 信号丢失 | 降级到视觉里程计 | 信号恢复 ≥ 30s 连续 |
| SLAM 定位置信度 < 阈值 | 降速 50% → 如继续下降则停止 | 重定位成功 |
| 坡度 > 25° | 立即停止 → 上报 | 操作员确认 |
| 电机扭矩异常（湿土沉降） | 停止 → 反扭矩尝试 → 上报 | 操作员评估 |
| 电池 < 15% | 终止当前行 → 返回充电站 | 充电至 ≥ 80% |
| 通信中断 > 1s | 停止 → 等待恢复 | 链路恢复且数据对齐 |
| 任一子系统故障 | 停止 → 诊断上报 | 故障清除 |

## 4. 技术约束

### 4.1 传感器

| 传感器 | 型号/规格 | 用途 |
|--------|----------|------|
| LiDAR | 待定 (≥20m 范围, ≥10Hz) | 障碍物检测, SLAM |
| 深度相机 | 待定 (RGB-D, 全局快门) | 行识别, 藤蔓检测 |
| RTK-GPS | 待定 (厘米级精度) | 全局定位 |
| IMU | 待定 (9-DOF) | 姿态估计, 坡度补偿 |

### 4.2 计算平台

| 层级 | 硬件 | 职责 |
|------|------|------|
| 边缘计算 | 待定 (ARM/x86, ≥8GB RAM) | 实时感知/导航/控制 |
| 云计算 (可选) | — | 离线建图, 数据分析 |

### 4.3 通信

- 内部总线: CAN / EtherCAT（确定性控制）
- 对外通信: WiFi / 4G / LoRa（遥测与指令）
- 协议: 待定（考虑 ROS 2 + DDS）

### 4.4 软件框架

- **框架**: ROS 2（机器人操作系统）
- **语言**: C++（实时关键路径）、Python（高层规划/数据分析）
- **中间件**: DDS（数据分发服务）
- **仿真**: Gazebo / Isaac Sim
- **容器化**: Docker（开发环境一致性）

### 4.5 代码原则

- **确定性优先**: 低层控制路径必须确定性执行，不可依赖概率输出
- **可观测性**: 每个子系统暴露诊断接口
- **传感器优先**: 世界状态以传感器融合结果为 truth，不信任纯推算
- **故障显式上报**: 子系统降级/异常 → 显式上报而非静默处理
- **无猜测执行**: 不确定性高的指令 → 停止并请求操作员确认

## 6. 开发流程

所有代码变更必须遵循 **Plan → Allocate → Implement → Review → Gate** 流程。

→ **[Agent 协作规范](AGENTS.md)** — Kanban 多 Agent 系统（Plan→Decompose→Dispatch→Review→Gate）

核心规则：
- 父 Agent（Hermes）负责架构设计、任务分配、代码审查、关卡判定
- 父 Agent **禁止编写任何 `src/` 下的代码**
- Hermes Worker Profile 负责代码实现，每个 Profile 只操作自己的子系统目录
- 安全审查（维度 A）和架构合规性（维度 B）为硬关卡，不通过则驳回
- ⚠️ Review 必须运行全部测试；未运行测试的审查无效
- ⚠️ 父 Agent 直接编写代码时，自写自审 = 盲区；必须运行测试并附结果

## 8. 通信与接口规范

### 8.1 操作员命令集

| 命令 | 参数 | 描述 |
|------|------|------|
| `MISSION_START` | zone_id | 启动指定区域作业 |
| `MISSION_PAUSE` | — | 暂停当前任务 |
| `MISSION_RESUME` | — | 恢复暂停的任务 |
| `MISSION_ABORT` | — | 中止当前任务（安全停车） |
| `E_STOP` | — | 紧急停止（最高优先级） |
| `RETURN_HOME` | — | 返回充电/停放点 |
| `SET_PARAM` | key, value | 设置运行参数 |
| `STATUS_REQ` | subsystem | 查询子系统状态 |

### 8.2 子系统间消息

所有子系统间通信使用强类型消息，通过 ROS 2 topic/service/action 实现。消息定义位于 `src/common/types/`。

## 9. 约束边界

### 绝对禁止

- ❌ 绕过运动控制子系统直接驱动执行器
- ❌ 在急停未释放状态下执行任何动作
- ❌ 禁用/忽略碰撞检测
- ❌ 在无人员检测覆盖区域自主作业
- ❌ 传感器故障时继续作业（除非已定义降级策略且经操作员批准）

### 设计约束

- 🔶 所有安全关键决策必须在边缘侧完成（不依赖云端）
- 🔶 直线行驶误差 ≤ ±5 cm（行内）
- 🔶 地头转向空间需求 ≤ 行宽的 1.5 倍
- 🔶 紧急停止响应时间 ≤ 100ms
- 🔶 系统启动自检时间 ≤ 30s
